# 클라이언트 복호화 실패 로그 개선

## 문제

클라이언트에서 복호화 실패 로그가 나오지만, 실제로는 서버에서 정상적으로 복호화되고 있어서 혼란을 줄 수 있습니다.

## 수정 내용

### 로그 메시지 개선
- "복호화 실패" → "클라이언트 복호화 실패, 서버로 암호화된 이름 전송"
- "암호화된 이름 반환 (서버에서 복호화 시도)" → "암호화된 이름을 서버로 전송 (서버에서 복호화 시도 예정)"

이렇게 하면:
1. 클라이언트에서 복호화가 실패했지만
2. 서버에서 복호화를 시도할 것임을 명확히 알 수 있습니다.
3. 서버에서 복호화가 성공하면 정상적으로 작동합니다.

## 수정된 로그 메시지

### Before
```
[발신자] 복호화 실패, 암호화된 이름 반환: user_id=..., 암호화된 이름="..."
[발신자] ❌ 복호화 실패 (모든 enc 후보 시도 완료): user_id=...
```

### After
```
[발신자] 클라이언트 복호화 실패, 서버로 암호화된 이름 전송 (서버에서 복호화 시도): user_id=..., 암호화된 이름="..."
[발신자] ❌ 클라이언트 복호화 실패 (모든 enc 후보 시도 완료), 서버로 암호화된 이름 전송: user_id=... (서버에서 복호화 시도 예정)
[발신자] 암호화된 이름을 서버로 전송 (서버에서 복호화 시도 예정): "..."
```

## 상태

✅ **수정 완료** - 클라이언트 로그가 더 명확해졌습니다.

## 참고

이것은 **정상적인 동작**입니다:
1. 클라이언트에서 복호화를 시도하지만 실패할 수 있음 (예: MY_USER_ID 불일치, enc 타입 불일치 등)
2. 클라이언트는 암호화된 이름을 서버로 전송
3. 서버에서 복호화를 시도하고 성공하면 정상 작동

이는 **fallback 메커니즘**으로 설계된 것이며, 클라이언트 복호화 실패가 문제가 되는 것은 아닙니다.
